# Этап 1: Проектирование

Дата фиксации этапа: 20 марта 2026  
Статус: `active`

## Цель этапа
Собрать дизайн и логику страниц в единый, согласованный проектный каркас до начала разработки функционала.

## Принципы этапа
1. Сначала структура и смысл блоков, потом визуальные детали.
2. Один экран решает одну задачу пользователя.
3. Один главный CTA на сценарий.
4. Функциональные гипотезы фиксируем, но не реализуем до отдельного этапа.

## Что входит в этап проектирования
1. Карта страниц и приоритетов.
2. Каркас главной страницы по блокам.
3. Базовый UI-контур: цвета, типографика, кнопки, карточки, сетка.
4. Контентный скелет для ключевых блоков.
5. Правила адаптивности для desktop/tablet/mobile.

## Что не входит в этот этап
1. Глубокая backend-логика.
2. Интеграции CRM, ботов, аналитики.
3. Полная автоматизация контента.
4. Сложные калькуляторы и нестандартные интерактивы.

## Артефакты этапа
- [ ] Согласована карта страниц.
- [ ] Согласован wireframe главной страницы.
- [ ] Согласован визуальный стиль (набор правил, не только референсы).
- [ ] Подготовлен контентный черновик ключевых блоков.
- [ ] Подготовлены требования к этапу функционала.

## Критерии готовности этапа
1. По каждому блоку главной страницы есть ответ на вопрос: зачем он нужен и к какому действию ведёт.
2. Есть единый стиль для компонентов: кнопки, карточки, заголовки, отступы.
3. Есть список функциональных задач, отложенных на этап 2.
4. Нет спорных мест формата "доделаем в процессе" по базовой структуре.

## Риски этапа и как закрываем
1. Риск: быстрый уход в функционал до фиксации структуры.  
Решение: все функциональные идеи пишем в backlog, но не реализуем.
2. Риск: визуальные правки без рамок.  
Решение: утверждаем стиль-пакет и меняем только по зафиксированным причинам.
3. Риск: перегрузка главной.  
Решение: приоритизация блоков по JTBD и одному главному сценарию.

## Выход этапа
Комплект согласованных проектных материалов, на основе которого можно начинать техническую реализацию без потери логики.
